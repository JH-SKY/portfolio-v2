수정 방향

단일 노드가 아니라 후속 노드로 이어지는 구조

Router가 질문 의도와 필요한 작업을 함께 판단하고, 각 노드의 책임 범위를 분리해 문제 발생 지점을 추적하기 쉽게 만들었습니다.

“하나의 LLM이 전부 답하게 만들 수도 있지만, 그러면 왜 그런 답이 나왔는지 추적하기 어렵다.”
Troubleshooting · BIZ-FUND-AI

사이드 프로젝트의 강점은 기능 설명이 아니라,
실제 장애를 복구한 기록입니다.

Issue 01

순환 참조

Hard Delete에서 Soft Delete로 바꾸며 AuthService와 BusinessService가 서로 참조하는 구조가 됐습니다.

Method Injection + Router Orchestration

서비스 간 직접 참조를 제거하고 Router가 두 도메인을 조율하도록 변경했습니다.

Issue 02

Rollback 데이터 유실

중복 키 에러를 잡으려고 rollback을 호출하자 이전 성공 데이터까지 사라졌습니다.

UPSERT + 트랜잭션 범위 재설계

중복을 예외가 아니라 정상 흐름으로 처리해 1,282건을 데이터 유실 없이 저장했습니다.

Issue 03

외부 API 파일 신뢰성

확장자와 실제 파일 형식이 달라 파싱 실패와 전체 롤백이 발생했습니다.

Magic Number + Savepoint

파일 앞부분 바이너리로 실제 형식을 감지하고, 실패 건만 롤백하도록 바꿨습니다.

Troubleshooting · Operating Reliability

운영 화면이 깨지고, DB 연결이 끊기고,
빈 데이터가 쌓이는 상황까지 복구했습니다.

Issue 05

Admin 로그인 500 + 다중 404

서비스 함수 파라미터명이 count에서 total_count로 바뀌었지만 호출부에 반영되지 않아 500이 발생했습니다. 동시에 프론트가 호출하는 Admin API 일부가 미구현되어 404가 연쇄 발생했습니다.

호출부 시그니처 정합성 복구
Feedback / Monitoring / Insights 라우트 추가
안전한 기본 응답으로 화면 크래시 차단
Issue 06

Supabase Timeout + 데이터 품질 오염

with_ai=True Full Sync에서 200~300건 이후 DB 연결이 끊겼고, Swagger 기본값을 놓쳐 ai_summary가 NULL인 1,255건이 저장됐습니다.

pool_pre_ping / pool_recycle / prepared_statement_cache_size 조정
페이지별 독립 세션 구조 도입
PARSE_ERROR / ANALYSIS_ERROR 저장 차단

03

프롬프트보다 구조를 먼저 의심

기능이 복잡해질수록 하나의 LLM에 몰아넣지 않고, 노드·도메인·트랜잭션 경계를 나눕니다.

04

다시 측정 가능한 상태로 만든다

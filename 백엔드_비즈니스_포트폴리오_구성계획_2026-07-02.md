## 2. 전체 프로젝트에서 뽑은 핵심 어필 포인트

### 2-1. 백엔드 면접관이 좋아할 포인트

#### 1. FastAPI + SQLAlchemy 2.0 Async 기반 도메인 구조

`biz-fund-ai`는 `auth`, `business`, `policy`, `chat`, `diagnosis`, `notification`, `admin` 등 도메인 단위로 라우터/서비스/레포지토리/스키마가 나뉘어 있다. 단순 CRUD가 아니라 서비스가 커질 때 변경 지점을 좁히는 구조다.

어필 문장:

> FastAPI, SQLAlchemy 2.0 Async, Pydantic v2 기반으로 도메인별 Router-Service-Repository 구조를 설계했고, 정책 추천/진단/채팅/관리자 기능을 분리해 확장 가능성을 확보했습니다.

근거:

- `biz-fund-ai/backend/src/app/api/v1/*_router.py`
- `biz-fund-ai/backend/src/app/domains/*/service.py`
- `biz-fund-ai/backend/src/app/domains/*/repository.py`
- `biz-fund-ai/docs/02_system_architecture.md`

#### 2. 트랜잭션 원자성 문제를 UPSERT와 Savepoint로 해결

`biz-fund-ai/docs/05_troubleshooting_log.md`의 Issue-02, Issue-03, Issue-06이 강력하다. 특히 "로그는 1,282건 성공인데 DB는 300여 건"이라는 문제를 실제 DB 결과와 비교해 찾아낸 점이 좋다.

어필 문장:

> 대량 정책 데이터 수집 중 단일 중복 에러가 전체 롤백으로 이어지는 문제를 분석하고, PostgreSQL `ON CONFLICT DO UPDATE` 기반 UPSERT와 `begin_nested()` Savepoint로 배치 멱등성과 실패 격리를 확보했습니다.

근거:

- `biz-fund-ai/docs/05_troubleshooting_log.md`
- `biz-fund-ai/backend/src/app/domains/policy/sync_service.py`
- 실제 결과: 약 300건 수준으로 머물던 DB 반영 건수를 1,282건까지 회복

#### 3. Supabase pgbouncer 특성을 반영한 DB 커넥션 설계

단순 DB 연결이 아니라 클라우드 DB 인프라 제약을 이해하고 코드에 반영했다. `pool_recycle=300`, `pool_pre_ping=True`, `prepared_statement_cache_size=0`, 페이지별 독립 세션 구조가 핵심이다.

어필 문장:

> Supabase pgbouncer 환경에서 장시간 AI 배치가 `ConnectionDoesNotExistError`로 중단되는 문제를 해결하기 위해 커넥션 재활용 주기와 prepared statement cache를 조정하고, 100건 단위 페이지별 독립 세션으로 배치 세션 수명을 제한했습니다.

근거:

- `biz-fund-ai/backend/src/app/database/postgres/database.py`
- `biz-fund-ai/docs/05_troubleshooting_log.md` Issue-06
- 실제 결과: AI 구조화 포함 1,255건 Full Sync 중단 없이 완료

#### 4. Hybrid RAG 검색 파이프라인 설계

pgvector만 쓴 것이 아니라 Vector Search, Elasticsearch BM25 + Nori, RRF를 결합했다. 특히 정책명/기관명은 sparse retrieval, 의미 편차는 vector retrieval로 해결했다는 판단 근거가 좋다.

어필 문장:

> 정책자금 검색에서 단일 벡터 검색이 고유명사와 기관명 매칭에 약한 문제를 해결하기 위해 pgvector cosine search와 Elasticsearch BM25 + Nori를 병렬로 수행하고, RRF(k=60)로 정책 단위 검색 결과를 융합했습니다.

근거:

- `biz-fund-ai/docs/06_rag_pipeline_design.md`
- `biz-fund-ai/backend/src/app/agents/biz_mong/tools/policy_rag.py`
- 최종 설계: vector top 20 + BM25 top 20 → RRF → top 5 정책 → 상위 3청크 답변 컨텍스트

#### 5. 테스트와 평가 하네스

`biz-fund-ai`에는 서비스 단위 테스트와 AI 품질 평가 스크립트가 있다. "AI 챗봇 만들었다"보다 "품질을 측정했다"가 훨씬 강하다.

어필 문장:

> 정책 추천 챗봇의 품질을 수동 확인에 의존하지 않고, 12개 시나리오 평가셋으로 라우팅 정확도, 예상 정책 적중, 핵심 키워드 포함 여부를 자동 검증하는 하네스를 만들었습니다.

근거:

- `biz-fund-ai/docs/04_experiment_and_test.md`
- `biz-fund-ai/backend/src/app/scripts/evaluate_bizmong_quality.py`
- 결과: 12개 중 10개 통과, 83.3%
- 비용 최적화 전후에도 83.3% 유지

#### 6. 인증/인가와 상태 정합성

`study-room-reservation`과 `biz-fund-ai` 모두 인증/인가 사례가 있다. 특히 로컬스토리지 온보딩 상태를 DB 기반 SSOT로 전환한 경험이 백엔드 면접에서 좋다.

어필 문장:

> 클라이언트 로컬스토리지 상태에 의존하던 온보딩 판단을 실제 DB의 활성 사업장 존재 여부 기준으로 바꿔 멀티디바이스 환경에서 인증/온보딩 정합성을 확보했습니다.

근거:

- `biz-fund-ai/docs/05_troubleshooting_log.md` Issue-04
- `biz-fund-ai/backend/src/app/domains/auth/repository.py`

#### 7. 예약 충돌 방지 알고리즘과 소유권 기반 인가

기본기 증명으로 좋다. `(기존 시작 < 새 종료) AND (기존 종료 > 새 시작)` 겹침 판정은 깔끔하고 설명하기 좋다.

어필 문장:

> 예약 시스템에서 시간 겹침을 SQL 조건으로 일반화해 방/사용자 중복 예약을 차단하고, 리뷰 작성은 실제 이용 완료 예약 소유자만 가능하도록 소유권 기반 인가를 적용했습니다.

근거:

- `study-room-reservation/app/repositories/reservation_repo.py`
- `study-room-reservation/app/services/reservation_service.py`
- `study-room-reservation/app/services/review_service.py`

#### 3. POC에서 검증 지표를 정의하는 능력

근거:

HR Copilot의 채용공고 컴플라이언스, 일괄등록 사전 선별 설계가 좋다. 특히 "AI가 자동 탈락시키지 않고 HR이 최종 선택"하게 한 부분은 비즈니스/윤리/운영 감각을 보여준다.

어필 문장:

> 문서 일괄등록 사전 선별 기능을 설계할 때 AI가 자동으로 탈락시키는 방식이 아니라, 추천 후보를 기본 선택으로 제안하되 최종 판단은 HR 담당자가 하는 UX를 권장했습니다.

#### 6. 고객 미팅에서 쓸 수 있는 "기술 번역" 소재

다음 표현들이 고객 대응에 좋다.

- RAG: "문서 검색과 답변 생성을 분리해서, 답변이 어떤 근거에서 나왔는지 추적 가능하게 만든 구조"
- 평가 하네스: "프롬프트를 바꿔도 품질이 무너지지 않는지 확인하는 회귀 테스트"
- UPSERT: "같은 데이터를 다시 넣어도 서비스 데이터가 꼬이지 않게 하는 재실행 안전장치"
- Savepoint: "공고 1개 실패가 전체 배치를 멈추지 않게 하는 실패 격리 장치"
- LLM 비용 로그: "AI 기능별 원가를 고객/운영자가 볼 수 있게 만드는 원가관리 장부"

## 3. 포트폴리오 1: 백엔드 개발자 지원용 구성안

### 3-1. 전체 방향

목표는 "AI도 할 줄 아는 백엔드 개발자"가 아니라 "운영 가능한 백엔드 시스템을 설계하고 장애를 해결할 수 있는 개발자"로 보이게 하는 것이다.

기존 에이전트 포트폴리오가 LangGraph와 RAG 중심이라면, 백엔드 포트폴리오는 다음 키워드를 전면에 세운다.

- FastAPI async backend
- domain architecture
- PostgreSQL / pgvector / SQLAlchemy 2.0
- transaction and idempotency
- batch reliability
- auth and authorization
- observability / cost log
- test and evaluation

### 3-2. 추천 페이지 수

PDF/HTML 기준 10~12페이지가 적당하다. 개발자 포트폴리오에서 너무 많은 페이지는 오히려 집중도가 떨어진다.

#### Page 03. Main Project: Biz-Fund-AI Architecture

메시지:

> 정책 추천 AI 서비스를 FastAPI + PostgreSQL + LangGraph + pgvector 기반으로 설계한 1인 프로젝트

내용:

#### Page 04. Transaction & Batch Reliability

메시지:

> 대량 정책 데이터 수집에서 가장 중요한 것은 "성공 로그"가 아니라 실제 DB 정합성입니다.

사례 1: UPSERT

- 문제: 중복 키 에러 후 rollback 때문에 성공 데이터까지 사라짐
- 해결: `ON CONFLICT DO UPDATE`
- 결과: DB 반영 약 300건 → 1,282건

사례 2: Savepoint

- 문제: 공고 1건 파싱 실패가 전체 배치 중단으로 확산
- 해결: `begin_nested()`로 공고별 실패 격리
- 결과: 실패 공고만 제외하고 배치 지속

사례 3: 품질 미달 데이터 저장 차단

- 문제: `with_ai=False` 기본값으로 AI 필드가 NULL인 데이터 1,255건 저장
- 해결: `PARSE_ERROR`, `ANALYSIS_ERROR`는 저장 제외
- 결과: RAG에 들어가는 데이터 품질 기준 확립

#### Page 05. Cloud DB / Supabase Connection Issue

메시지:

> 로컬에서 문제없던 배치도 클라우드 DB의 연결 정책을 만나면 장애가 됩니다.

구성:

- 장애: `asyncpg.exceptions.ConnectionDoesNotExistError`
- 원인:
  - Supabase pgbouncer idle timeout
  - 장수 세션
  - prepared statement cache 충돌
- 해결:
  - `pool_pre_ping=True`
  - `pool_recycle=300`
  - `prepared_statement_cache_size=0`
  - 페이지별 독립 세션
- 결과:
  - AI 구조화 포함 1,255건 Full Sync 완료

이 페이지는 백엔드 면접관에게 매우 강하다. 인프라와 애플리케이션 코드의 경계를 이해했다는 신호다.

#### Page 06. Hybrid RAG Search Backend

메시지:

> 정책 검색은 의미 검색만으로 부족해, 고유명사 검색과 의미 검색을 결합했습니다.

## 9. 포트폴리오 문장 재료

### 백엔드용 핵심 문장

- "성공 로그가 아니라 실제 DB 반영 결과를 기준으로 배치 안정성을 검증했습니다."
- "중복 데이터를 예외로 처리하지 않고 UPSERT라는 정상 흐름으로 설계했습니다."
- "AI 호출이 긴 배치에서는 세션 수명을 페이지 단위로 제한해 클라우드 DB timeout을 회피했습니다."
- "검색 품질을 위해 vector search와 sparse retrieval을 분리하고 RRF로 결합했습니다."
- "온보딩 완료 여부는 클라이언트 상태가 아니라 서버 DB를 단일 진실 원천으로 삼았습니다."
- "예약/리뷰 같은 기본 도메인에서도 시간 충돌, 소유권, 기한 검증을 명확히 코드화했습니다."

### 비즈니스 엔지니어용 핵심 문장

- "고객의 막연한 불편을 사용자 여정과 기능 요구사항으로 번역했습니다."
- "PoC 성공 기준을 데모 감각이 아니라 고정 평가셋과 루브릭으로 정의했습니다."
- "AI가 판단을 대체하지 않고 담당자의 결정을 보조하도록 human-in-the-loop UX를 설계했습니다."
- "비용은 운영 이후에 보는 숫자가 아니라 설계 단계에서 통제해야 할 변수로 봤습니다."
- "고객에게 설명 가능한 AI를 위해 답변 근거, 사용량, 실패 상태를 기록했습니다."
- "기술 구현과 고객 업무 흐름 사이를 연결하는 역할에 강점이 있습니다."

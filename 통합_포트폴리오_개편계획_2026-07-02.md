# 한 HTML 안에 3개의 독립 포트폴리오를 넣는 개편 계획

작성일: 2026-07-02

## 1. 다시 잡은 기준

이번 작업은 페이지 수를 먼저 정해놓고 내용을 억지로 끼워 넣는 작업이 아니다.

정확한 기준은 이렇다.

- 에이전트 개발자용 포트폴리오는 이미 만들어둔 현재 포트폴리오를 유지한다.
- 백엔드 개발자용 포트폴리오는 `BIZ-FUND-AI`에서 쓸 수 있는 내용을 전부 꺼낸 뒤 페이지를 만든다.
- 솔루션엔지니어용 포트폴리오도 `BIZ-FUND-AI`에서 비즈니스, 문제정의, 사용자 여정, 운영 구조를 전부 꺼낸 뒤 페이지를 만든다.
- 최종 페이지 수는 먼저 정하지 않는다.
- 내용을 전부 펼친 뒤, 자연스럽게 몇 장이 필요한지 결정한다.

즉 `한 HTML`은 파일 관리 방식일 뿐이다.
실제로는 그 안에 직무별 포트폴리오 3개가 들어간다.

## 2. 절대 건드리지 않을 부분

### 에이전트 개발자용 포트폴리오

현재 `portfolio-v2/index.html`에 만들어진 에이전트 중심 포트폴리오는 이미 완성된 자산으로 본다.

따라서 원칙은 아래다.

- 기존 에이전트 포트폴리오의 흐름은 건드리지 않는다.
- 기존 에이전트 포트폴리오를 줄이지 않는다.
- 기존 에이전트 포트폴리오를 백엔드/솔루션 내용에 맞춰 섞지 않는다.
- 한 HTML 안에서 앞쪽 구간으로 그대로 둔다.

현재 에이전트 포트폴리오에는 이미 아래 내용이 들어가 있다.

- 표지
- 프로젝트 맵
- `HR Copilot` 개요
- `HR Copilot` 문제정의와 구조
- LangSmith Trace 기반 분석
- 품질 평가와 개선 결과
- `BIZ-FUND-AI`의 RAG, 비용, 데이터 운영 일부
- 클로징

이 부분은 “기존 완성본”으로 취급한다.
새로 기획할 대상은 주로 `백엔드용 포트폴리오`와 `솔루션엔지니어용 포트폴리오`다.

## 3. 전체 HTML 배치 방식

HTML은 하나만 둔다.
다만 내부 구조는 이렇게 간다.

- 에이전트 개발자용 포트폴리오: 기존 완성본 유지
- 백엔드 개발자용 포트폴리오: 기존 뒤에 새로 추가
- 솔루션엔지니어용 포트폴리오: 백엔드 뒤에 새로 추가

PDF 제출 방식:

- 에이전트 지원 시: 에이전트 구간만 출력
- 백엔드 지원 시: 백엔드 구간만 출력
- 솔루션엔지니어 지원 시: 솔루션 구간만 출력

따라서 전체 HTML이 길어지는 것은 문제가 아니다.
오히려 각 구간이 직무별 제출본으로 충분히 완성되어야 한다.

## 4. 페이지 수를 정하는 방식

페이지 수를 먼저 정하지 않는다.

작업 순서는 아래처럼 간다.

1. `BIZ-FUND-AI`에서 백엔드용으로 쓸 수 있는 내용을 전부 뽑는다.
2. `BIZ-FUND-AI`에서 솔루션엔지니어용으로 쓸 수 있는 내용을 전부 뽑는다.
3. 비슷한 내용끼리 묶는다.
4. 한 묶음이 한 페이지로 충분하면 한 페이지로 만든다.
5. 내용이 많으면 자연스럽게 두 페이지로 나눈다.
6. 그렇게 나온 결과가 최종 페이지 수다.

즉 페이지 수는 결과다.
출발점이 아니다.

## 5. 면접관 관점에서 진짜 앞세울 것

아래는 단순 기능 목록이 아니라,
백엔드/FDE/솔루션엔지니어 면접관 입장에서 "이 사람 꽤 깊게 해봤네"라고 읽힐 가능성이 높은 것들이다.

모든 기능을 같은 비중으로 펼치면 오히려 약해진다.
그래서 실제 페이지를 만들 때는 아래 우선순위를 기준으로 잡는다.

## 5-1. 백엔드 면접관에게 가장 강한 A급 소재

### A1. 정책 데이터 파이프라인의 운영 안정성

왜 매력적인가:

- 단순 CRUD나 챗봇보다 백엔드다운 문제다.
- 외부 API, 파일 다운로드, 파싱, AI 구조화, DB 저장, 임베딩, 검색 인덱싱까지 긴 파이프라인을 다뤘다.
- 실패가 날 수밖에 없는 구간에서 장애 범위를 줄이고 재실행 가능하게 만든 경험이 드러난다.

반드시 보여줄 근거:

- `PolicySyncAgent`
- 파일 시그니처 기반 실제 파일 형식 판별
- `savepoint` 기반 개별 공고 실패 격리
- `upsert` 기반 멱등성
- 페이지 단위 DB 세션 분리
- PostgreSQL Advisory Lock 기반 스케줄러 중복 실행 방지
- `batch_logs`에 API/파싱/AI/DB 실패 원인 분리

면접관이 읽는 역량:

- 배치 안정성
- 트랜잭션 이해
- 외부 데이터 품질 문제 대응
- 장애 격리
- 운영 가능한 백엔드 설계

이건 백엔드 포트폴리오에서 반드시 단독 페이지 또는 2페이지로 크게 잡는다.

### A2. 트러블슈팅 수치가 있는 데이터 적재 개선

왜 매력적인가:

- "구현했다"가 아니라 "망가졌고, 원인을 찾고, 수치로 개선했다"는 이야기다.
- 면접에서 가장 말하기 좋은 소재다.

반드시 보여줄 근거:

- 성공 로그는 1,282건인데 실제 DB에는 약 300건만 남던 문제
- rollback으로 이전 성공 데이터까지 날아가던 문제
- UPSERT 전환 후 1,282건 저장 가능
- 파일 선택/판별 개선으로 파싱 성공률 약 24%에서 57%로 개선
- `with_ai=True` 장기 배치 1,255건에서 DB 연결 timeout을 세션 수명 분리로 해결

면접관이 읽는 역량:

- 로그와 실제 DB 상태를 대조하는 검증 습관
- DB 트랜잭션 이해
- 클라우드 DB/pgbouncer 환경 문제 대응
- 장기 실행 작업 설계

이건 백엔드 포트폴리오의 하이라이트 페이지로 써야 한다.

### A3. Hybrid RAG를 서비스 요구에 맞게 설계한 점

왜 매력적인가:

- 그냥 벡터 검색을 붙인 게 아니라 한국어 정책 검색 문제를 이해하고 검색 구조를 설계했다.
- 백엔드 면접관에게도 검색/데이터 설계 역량으로 읽힌다.

반드시 보여줄 근거:

- pgvector 기반 의미 검색
- Elasticsearch BM25 기반 sparse retrieval
- Nori 분석기
- RRF 결합
- PostgreSQL `ilike` fallback
- 정책명/기관명 같은 고유명사 검색 누락 문제
- 섹션 기반 청킹, 800자 window fallback, contextual embedding

면접관이 읽는 역량:

- 요구사항에 맞는 검색 기술 선택
- 한국어 검색 특성 이해
- fallback 설계
- RAG 품질을 데이터 파이프라인과 연결해서 보는 관점

이건 "AI" 페이지가 아니라 "검색 백엔드 설계" 페이지로 써야 한다.

### A4. 진단/시뮬레이션을 채팅 에이전트와 분리한 설계

왜 매력적인가:

- 모든 걸 LLM agent에 몰아넣지 않고, 규칙 기반 판단과 상태 저장이 필요한 기능을 REST API로 분리했다.
- 백엔드 설계 판단이 보인다.

반드시 보여줄 근거:

- `/diagnoses`
- `/simulations`
- rule_engine
- 재무건전성 40%, 성장잠재력 20%, 운영안정성 25%, 리스크관리 15%
- RED/YELLOW/GREEN 판정
- simulation_logs 저장
- 조건 변화에 따른 점수 변화 계산

면접관이 읽는 역량:

- 책임 분리
- LLM 남용 방지
- 비즈니스 룰 모델링
- 추적 가능한 의사결정 로직

이건 백엔드 포트폴리오에서 "도메인 로직 설계" 페이지로 잡는다.

### A5. 운영자/관리자 기능과 관측성

왜 매력적인가:

- 포트폴리오 프로젝트에서 운영자 관점까지 만든 사람은 많지 않다.
- 서비스 운영까지 생각했다는 신호가 된다.

반드시 보여줄 근거:

- 관리자 로그인과 관리자 JWT
- 정책 등록/수정
- 콘텐츠 발행
- AI 상담 로그 모니터링
- 배치 상태 조회
- 배치 상세 로그
- 진단/시뮬레이션 로그 모니터링
- 감사 로그
- agent run/node log
- p50/p95 latency
- 모델별 cost 집계

면접관이 읽는 역량:

- 운영 백오피스 설계
- 관측성
- 비용 모니터링
- CS/품질 개선 루프 고려

이건 백엔드와 FDE 양쪽에 모두 쓸 수 있는 강한 소재다.

### A6. 비용 최적화와 평가 하네스

왜 매력적인가:

- AI 기능의 비용을 줄였는데, 품질 유지 여부까지 검증했다.
- "싸게 바꿨다"가 아니라 "회귀 검증까지 했다"가 핵심이다.

반드시 보여줄 근거:

- Router: GPT-4o에서 GPT-4o-mini
- Greeting: 정적 응답
- RAG: GPT-4o-mini
- PolicySync 구조화: GPT-4o 유지
- 약 15배 비용 절감
- 평가 하네스 12개 케이스 중 10개 통과
- 최적화 전후 83.3% 유지

면접관이 읽는 역량:

- 비용 의식
- 작업 복잡도별 모델 선택
- 회귀 테스트
- AI 기능을 운영 비용까지 고려해 설계하는 태도

이건 백엔드보다는 "AI 포함 백엔드", FDE, 솔루션 쪽에서도 강하게 쓸 수 있다.

## 5-2. 백엔드에서 B급으로 묶어 보여줄 소재

아래는 중요하지만 단독 페이지로 너무 길게 잡으면 평범해질 수 있다.
상위 A급 소재를 보조하는 구조로 묶는 게 좋다.

- FastAPI router/service/repository/model/schema 구조
- 인증과 Refresh Token
- 사업장 CRUD
- 정책 목록/상세/북마크 API
- 알림
- 비즈픽 콘텐츠
- 기본 테이블 목록
- 테스트 목록

이 내용들은 `전체 서비스 구조`, `도메인 계층`, `API 범위`, `검증 체계` 같은 페이지 안에 묶어서 쓰는 편이 좋다.

## 5-3. FDE/솔루션 면접관에게 가장 강한 A급 소재

### F1. 고객 문제를 서비스 여정으로 번역한 점

왜 매력적인가:

- FDE/솔루션엔지니어는 기술만 아는 사람보다 문제를 구조화하고 고객 언어로 설명할 수 있는 사람이 강하다.
- `BIZ-FUND-AI`는 5단계 사용자 여정이 이미 있다.

반드시 보여줄 근거:

- 정책자금 최초 신청 소상공인
- 전략적 개선을 원하는 소상공인
- Biz-On
- Biz-Check
- Business Optimization
- Biz-Benchmark
- Retention & Support

면접관이 읽는 역량:

- 고객 문제 정의
- 사용자 여정 설계
- 기능 우선순위 판단
- 기술을 고객 행동 흐름으로 바꾸는 능력

솔루션/FDE 포트폴리오의 첫 핵심 페이지로 둔다.

### F2. "검색"이 아니라 "진단과 다음 행동"으로 설계한 점

왜 매력적인가:

- 단순 정책 검색 서비스라면 흔하다.
- 이 프로젝트는 사용자의 상태를 진단하고, 조건 변화 시뮬레이션으로 다음 행동까지 제안한다.

반드시 보여줄 근거:

- 정밀 진단
- RED/YELLOW/GREEN
- 직원 채용 시 점수 변화
- 특허/벤처/여성기업 가점
- 부채비율 개선
- 정책 수혜 가능성 변화

면접관이 읽는 역량:

- 고객 의사결정 지원
- 업무 흐름 이해
- 도메인 규칙을 솔루션 기능으로 바꾸는 능력

이건 FDE/솔루션용에서 가장 강하게 앞세울 페이지다.

### F3. 운영자/관리자까지 고려한 점

왜 매력적인가:

- 고객에게 보여주는 화면만 만든 게 아니라, 운영자가 서비스를 관리할 수 있는 구조를 설계했다.
- 실제 도입/운영을 생각했다는 증거다.

반드시 보여줄 근거:

- 관리자 정책 등록/수정
- 콘텐츠 발행
- 배치 상태 확인
- AI 상담 로그 모니터링
- 진단/시뮬레이션 로그
- 감사 로그
- correction notes

면접관이 읽는 역량:

- 운영 시나리오 이해
- 고객사 도입 후 관리 관점
- 장애/품질 대응 프로세스 고려

FDE 관점에서는 백엔드의 관리자 기능을 "운영 설계"로 번역해서 보여준다.

### F4. 이해관계자별 가치와 확장 방향

왜 매력적인가:

- 솔루션엔지니어는 고객과 회사 양쪽 가치를 설명해야 한다.
- `BIZ-FUND-AI`는 B2C/B2B 확장 서사가 있다.

반드시 보여줄 근거:

- 소상공인: 정책 탐색 시간 절감
- 스마트기기 업체: ROI 시뮬레이션 기반 매칭
- 기관/지자체: 정책 효과 분석 API
- 금융기관: 보증/정책상품 연계
- 파트너 상담 연결
- 심화 분석 리포트

면접관이 읽는 역량:

- 비즈니스 모델 이해
- 고객 가치 설명
- 확장 가능성 판단
- 기술과 사업 사이 연결

이건 솔루션/FDE 포트폴리오의 후반부 핵심 페이지로 둔다.

### F5. 기술 의사결정을 고객 언어로 설명할 수 있는 점

왜 매력적인가:

- FDE는 내부 개발자와 외부 고객 사이를 연결해야 한다.
- "왜 Hybrid RAG냐", "왜 진단은 REST API냐", "왜 평가 하네스가 필요하냐"를 고객 가치로 번역할 수 있어야 한다.

번역 예시:

- Hybrid RAG: 정책명과 기관명은 정확히 찾고, 생활어 질문은 의미로 찾기 위해 필요했다.
- 진단/시뮬레이션 분리: 상담 답변과 점수 계산은 신뢰 기준이 다르기 때문에 분리했다.
- 평가 하네스: AI 답변을 감으로 운영하지 않고 반복 검증하기 위해 만들었다.
- 관리자 모니터링: 운영 중 실패 케이스를 찾아 다음 개선으로 연결하기 위해 만들었다.

면접관이 읽는 역량:

- 기술 설명력
- 고객 커뮤니케이션
- 구현 의사결정의 비즈니스 연결

이건 FDE 포트폴리오에서 아주 매력적인 차별점이다.

## 5-4. FDE/솔루션에서 뒤로 빼도 되는 소재

아래는 FDE 포트폴리오에서 너무 앞에 나오면 메시지가 흐려진다.

- 세부 DB 테이블 전체 목록
- router/service/repository 구조 세부
- LangGraph 노드 상세
- 임베딩 차원 같은 구현 세부
- 테스트 파일 목록

이 내용들은 FDE용에서는 "기술 근거" 정도로 짧게만 쓰고,
백엔드용에서 자세히 보여주는 것이 낫다.

## 5-5. 최종 선별 결론

백엔드 포트폴리오에서 가장 앞세울 것은 아래 순서다.

1. 정책 데이터 파이프라인 운영 안정성
2. 트러블슈팅 수치가 있는 데이터 적재 개선
3. Hybrid RAG 검색 백엔드 설계
4. 진단/시뮬레이션 도메인 로직 분리
5. 관리자/운영/관측성
6. 비용 최적화와 평가 하네스
7. API/도메인/DB 구조

FDE/솔루션 포트폴리오에서 가장 앞세울 것은 아래 순서다.

1. 고객 문제와 사용자 여정
2. 검색을 진단과 다음 행동으로 확장한 솔루션 설계
3. 운영자/관리자 관점
4. 이해관계자별 가치와 확장 방향
5. 기술 의사결정을 고객 가치로 번역하는 능력

이 순서가 실제 면접관 입장에서 가장 매력적으로 읽힐 가능성이 높다.

## 9. 실제 HTML 구현 기준

구현할 때 기준은 이렇다.

- 기존 에이전트 포트폴리오 구간은 유지한다.
- 그 뒤에 백엔드 포트폴리오 구간을 새로 붙인다.
- 그 뒤에 솔루션엔지니어 포트폴리오 구간을 새로 붙인다.
- 페이지 번호는 전체 HTML 기준으로 이어져도 된다.
- 대신 각 구간의 표지와 라벨은 명확히 다르게 둔다.

구간 라벨 예시:

- `AI Agent Developer Portfolio`
- `Backend Developer Portfolio`
- `Solution Engineer Portfolio`

중요한 점:

- 백엔드 구간을 만들 때 에이전트 포폴을 줄이지 않는다.
- 솔루션 구간을 만들 때 백엔드 포폴을 줄이지 않는다.
- 전체 장수가 늘어나는 것은 정상이다.

## 10. 다음 작업 순서

다음 실제 작업 순서는 아래가 맞다.

1. 현재 `index.html`의 에이전트 포트폴리오를 보존 구간으로 표시한다.
2. `BIZ-FUND-AI` 백엔드용 페이지 후보를 위 목록 기준으로 HTML 페이지로 하나씩 만든다.
3. 실제 작성하면서 한 페이지에 들어갈 분량이면 합치고, 넘치면 나눈다.
4. 백엔드 구간이 끝난 뒤 솔루션엔지니어 구간을 같은 방식으로 만든다.
5. 마지막에 PDF 출력용 페이지 범위 안내를 따로 정리한다.

이 계획의 핵심은 한 줄이다.

`페이지 수를 정해서 내용을 줄이는 게 아니라, 쓸 수 있는 내용을 전부 꺼낸 뒤 페이지가 따라오게 만든다.`

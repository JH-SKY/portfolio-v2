# 통합 HTML 기반 3종 포트폴리오 개편 계획

작성일: 2026-07-02

## 1. 수정된 이해

이번 작업의 핵심은 `3개의 직무용 포트폴리오를 만든다`는 것이다.
다만 파일은 `index1.html`, `index2.html`, `index3.html`처럼 쪼개지 않는다.

정확한 목표는 아래와 같다.

- `하나의 HTML` 안에
- `에이전트 개발자용 포트폴리오`
- `백엔드 개발자용 포트폴리오`
- `솔루션엔지니어용 포트폴리오`
- 이 3개를 `서로 섞지 않고`, `연속된 페이지 구간`으로 각각 완결되게 넣는다.

즉 구조는 이런 형태다.

- 1~N페이지: 에이전트 개발자용 포트폴리오
- N+1~M페이지: 백엔드 개발자용 포트폴리오
- M+1~K페이지: 솔루션엔지니어용 포트폴리오

그리고 PDF 제출 시에는 사용자가 페이지 범위만 골라서 각 직무용 포트폴리오처럼 제출한다.

이게 이번 설계의 가장 중요한 전제다.

## 2. 왜 이 방식이 맞는가

이 방식의 장점은 명확하다.

- HTML 유지보수는 한 파일만 하면 된다.
- 디자인 시스템과 공통 스타일을 재사용할 수 있다.
- 직무별로 메시지를 완전히 다르게 가져갈 수 있다.
- 제출할 때는 페이지 범위만 잘라서 사실상 `3개의 포트폴리오`처럼 쓸 수 있다.

반대로 이전처럼 `한 포트폴리오 안에 3개 직무를 섞어 보여주는 방식`은 이번 목적에 맞지 않는다.

왜냐하면:

- 채용 담당자는 보통 자기 직무에 맞는 서사만 빠르게 보고 싶다.
- 백엔드 지원용 문서에 에이전트 실험 페이지가 앞쪽에 길게 나오면 집중도가 떨어진다.
- 솔루션엔지니어용 문서도 백엔드 구현 상세가 너무 앞에 나오면 메시지가 흐려진다.

따라서 이번에는 `한 HTML`, `세 개의 독립 포트폴리오 구간`으로 가야 한다.

## 3. 실제 프로젝트를 보고 내린 판단

### 3-1. 에이전트 개발자용 포트폴리오 재료

가장 강한 재료는 `HR Copilot`이다.

근거:

- LangGraph 기반 멀티에이전트 구조
- `verification_point_extractor`를 포함한 품질 개선 구조
- 품질 평가 결과 `78.6 -> 91.8`
- LangSmith trace 및 실험 기반 개선 스토리
- 질문 생성 품질을 구조적으로 끌어올린 경험

보조 재료로 `BIZ-FUND-AI`의 BizMong도 일부 넣을 수 있다.

근거:

- Router / Chitchat / RAG / Stats 구조
- 평가 하네스 기반 품질 검증
- 작업별 모델 분리
- Hybrid RAG 파이프라인

결론:

- 에이전트 개발자용 포트폴리오는 지금 있는 `agent 중심 포폴`의 뼈대를 많이 유지해도 된다.
- 다만 `HR Copilot 중심 + BIZ-FUND-AI 보조` 구도로 정리하는 편이 더 선명하다.

### 3-2. 백엔드 개발자용 포트폴리오 재료

백엔드는 `BIZ-FUND-AI`를 메인으로 가져가는 게 맞다.

이유:

- 직접 기획
- 직접 설계
- 직접 구현
- API, DB, 배치, 운영 구조까지 전부 설명 가능

실제 근거:

- `docs/01_concept_and_requirements.md`
- `docs/02_system_architecture.md`
- `docs/03_data_design_spec.md`
- `docs/api_spec/*.md`
- `backend/src/app/api/v1/*`
- `backend/src/app/domains/*`
- `backend/src/app/agents/*`
- `backend/tests/*`

백엔드 포트폴리오에서 말할 수 있는 포인트:

- FastAPI 기반 API 설계
- 도메인 분리 구조
- PostgreSQL + pgvector
- Elasticsearch + Nori
- 진단/시뮬레이션 서비스 분리
- savepoint 기반 장애 격리
- upsert 기반 멱등 처리
- 배치 로그 / 운영 로그 / 관리자 기능

결론:

- 백엔드용 포트폴리오는 `BIZ-FUND-AI 단독 메인 프로젝트`로 가는 것이 가장 설득력 있다.

### 3-3. 솔루션엔지니어용 포트폴리오 재료

이 부분은 처음엔 약할 수 있다고 봤지만, 완전히 새로 만들 필요까지는 없다.

핵심은 `BIZ-FUND-AI`를 기술 프로젝트가 아니라 `문제 해결형 서비스 설계 사례`로 다시 읽는 것이다.

쓸 수 있는 포인트:

- 소상공인 사용자 문제 정의
- 5단계 사용자 여정
- 사업장 온보딩
- 정책 탐색 / 진단 / 시뮬레이션 / 비교 / 사후관리
- 관리자 대시보드
- 배치 운영
- B2C / B2B 확장 방향
- 이해관계자별 가치 제안

다만 한계도 있다.

- 고객사 도입 사례
- 현업 미팅 기록
- 실제 영업/프리세일즈 협업 자료
- 외부 PoC 제안서

이런 전형적 솔루션엔지니어 재료는 약하다.

그래서 결론은 이렇다.

- 솔루션엔지니어 포트폴리오를 `없애지는 않는다`.
- 하지만 `백엔드만큼 길게` 만들지도 않는다.
- `BIZ-FUND-AI를 문제정의/서비스설계/운영설계 관점으로 재서술한 짧고 선명한 포트폴리오`로 만든다.

## 4. 최종 권장 구조

총 3개의 포트폴리오를 한 HTML에 연속 배치한다.

추천 순서:

1. 에이전트 개발자용 포트폴리오
2. 백엔드 개발자용 포트폴리오
3. 솔루션엔지니어용 포트폴리오

이 순서를 추천하는 이유:

- 현재 자산 재활용이 쉽다.
- 첫 구간에서 기존 agent 포폴 구조를 많이 유지할 수 있다.
- 두 번째 구간에서 `BIZ-FUND-AI`를 가장 강하게 전개할 수 있다.
- 세 번째 구간은 앞선 프로젝트 자산을 재활용해 짧게 마무리하기 좋다.

## 5. 페이지 수 권장안

### 권장안 A

- 1~6페이지: 에이전트 개발자용
- 7~12페이지: 백엔드 개발자용
- 13~16페이지: 솔루션엔지니어용

총 16페이지

이 안이 가장 안정적이다.

이유:

- 각 직무용 포트폴리오가 `최소한의 완결성`을 가진다.
- 에이전트와 백엔드는 각각 6페이지 정도면 충분히 설득력 있게 구성 가능하다.
- 솔루션엔지니어는 4페이지면 과하지 않으면서도 독립 포트폴리오처럼 보인다.

### 권장안 B

- 1~5페이지: 에이전트 개발자용
- 6~11페이지: 백엔드 개발자용
- 12~14페이지: 솔루션엔지니어용

총 14페이지

이 안은 조금 더 압축형이다.
만약 전체 길이를 줄이고 싶다면 이쪽도 가능하다.

### 최종 추천

지금 상태에서는 `16페이지 안`을 먼저 기준으로 설계하고,
완성 후 필요하면 14페이지로 줄이는 것이 좋다.

- 관리자 대시보드
- 감사 로그
- 향후 B2B 확장 가능성
- 이해관계자별 가치

결론 메시지:

- 단순히 기능을 구현한 것이 아니라
- 도입 이후 운영과 확장까지 생각한 구조라는 점을 보여준다.

## 7. 실제 구현 시 디자인 전략

파일은 하나지만, 보는 사람은 `3개의 다른 포트폴리오`처럼 느껴야 한다.

그래서 구간별 톤을 약간 분리하는 것이 좋다.

### 에이전트 구간

- 현재 디자인 자산을 많이 재사용
- 블루/라임 계열 유지 가능
- 실험 / 품질 / 추적의 느낌 강화

### 백엔드 구간

- 네이비 / 그레이 / 화이트 기반
- 구조도, 표, 데이터 흐름 중심
- 더 단정하고 시스템 설계 문서 같은 느낌

### 솔루션엔지니어 구간

- 베이지 / 그린 / 딥블루 조합 가능
- 사용자 여정, 가치, 운영 흐름 중심
- 약간 제안서/컨설팅 문서 같은 인상

## 8. 무엇을 재사용하고 무엇을 새로 만들지

### 거의 그대로 재사용 가능한 것

- 현재 에이전트용 표지 레이아웃
- HR Copilot 구조 페이지의 일부
- BIZ-FUND-AI RAG / 비용 / 운영 페이지의 일부
- 카드형 KPI 표현
- 전체 A4 가로 페이지 시스템

### 수정해서 재사용할 것

- BIZ-FUND-AI 소개 페이지
- 클로징 페이지
- 각 섹션의 타이틀/카피

### 새로 만들어야 할 것

- 백엔드용 별도 표지
- 솔루션엔지니어용 별도 표지
- 솔루션엔지니어용 사용자 여정 페이지
- 솔루션엔지니어용 가치 제안 페이지
- 구간 시작 시 `이 페이지 범위는 어떤 포트폴리오인지` 명확히 보여주는 페이지 넘버/라벨 시스템

## 9. 최종 판단

### 에이전트 개발자용

유지 가치가 매우 높다.
현재 포트폴리오 자산을 가장 많이 살릴 수 있다.

### 백엔드 개발자용

`BIZ-FUND-AI`를 메인으로 충분히 강하게 만들 수 있다.
오히려 이번 개편에서 가장 경쟁력이 높아질 가능성이 크다.

### 솔루션엔지니어용

완전히 버릴 필요는 없다.
다만 `별도 대표 프로젝트`가 부족하므로
`BIZ-FUND-AI를 문제 해결형 서비스 사례로 재서술한 압축형 포트폴리오`로 가야 한다.

즉 최종 결론은:

- `3개 다 만든다`
- `한 HTML 안에 넣는다`
- `각 직무용 포트폴리오는 페이지 구간으로 완전히 분리한다`
- `에이전트와 백엔드는 완성도 높게`
- `솔루션엔지니어는 짧지만 명확하게`

## 10. 다음 실제 작업 순서

이후 실제 HTML 수정은 아래 순서가 가장 좋다.

1. 전체 페이지 수를 16페이지 기준으로 먼저 확정
2. `1~6`, `7~12`, `13~16` 구간 라벨 설계
3. 에이전트 구간 먼저 정리
4. 백엔드 구간 신설/강화
5. 솔루션엔지니어 구간 신설
6. 마지막으로 페이지 번호, PDF 출력 시 구간 식별 문구 정리

가장 중요한 구현 원칙은 하나다.

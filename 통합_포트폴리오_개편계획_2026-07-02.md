# 통합 포트폴리오 개편 계획

작성일: 2026-07-02

## 1. 이번 개편의 핵심 방향

현재 `portfolio-v2/index.html`은 사실상 `AI Agent Developer` 포지션에 맞춘 11페이지 구성이다.
특히 현재 포트폴리오는 다음 특징이 강하다.

- 표지부터 `LLM 품질을 수치로 개선한 AI 개발자` 메시지가 전면에 나온다.
- `HR Copilot`은 멀티에이전트 실험과 품질 개선 중심으로 4페이지를 사용한다.
- `BIZ-FUND-AI`는 Hybrid RAG, 비용 최적화, 데이터 적재 운영 중심으로 4페이지를 사용한다.

이번 개편에서는 이 포트폴리오를 완전히 새로 만드는 것이 아니라,
`하나의 HTML 안에서 3개 직무 관점이 공존하는 구조`로 재편하는 것이 맞다.

정리하면 목표는 아래와 같다.

- `에이전트 개발자`: 지금 강점이 가장 잘 드러나는 축이므로 기존 강점을 최대한 유지
- `일반 백엔드 개발자`: `BIZ-FUND-AI`를 메인 프로젝트로 재배치
- `솔루션엔지니어 / 컨설턴트 / FDE 계열`: 별도 프로젝트를 억지로 만들지 말고 `BIZ-FUND-AI` 안의 기획, 문제정의, 운영, 사용자 시나리오, 관리자 관점을 뽑아서 재구성

결론부터 말하면 `에이전트 개발자로 통일`하는 방향보다는,
`세 직무를 모두 보여주되 비중은 다르게 가져가는 방향`이 훨씬 좋다.

이유는 명확하다.

- 사용자 요청 자체가 이미 3트랙 분리를 전제로 한다.
- `BIZ-FUND-AI`는 실제로 기획 문서, 요구사항, API 명세, DB 설계, 운영 로그, 관리자 기능까지 있어서 백엔드/솔루션 관점 근거가 충분하다.
- 반대로 솔루션엔지니어 트랙만 따로 떼면 양이 약할 수 있으므로, `독립 직무 서사`로 만들되 `페이지 수는 적게` 가져가는 것이 현실적이다.

## 2. 실제 프로젝트를 보고 내린 판단

### 2-1. 현재 포트폴리오에서 유지할 자산

현재 `portfolio-v2/index.html` 기준:

- 페이지 1: 표지
- 페이지 2: 프로젝트 맵
- 페이지 3~6: `HR Copilot`
- 페이지 7~10: `BIZ-FUND-AI`
- 페이지 11: 클로징

특히 아래 요소는 그대로 살릴 가치가 높다.

- `HR Copilot`의 품질 개선 수치: 91.8점
- `BIZ-FUND-AI`의 비용 최적화: 약 15배 절감
- 평가 하네스 기반 검증: 12개 케이스 중 10개 통과
- `LangGraph`, `RAG`, `추적-개선-재검증`이라는 전체 메시지

즉, 기존 포트폴리오의 소재는 버릴 것이 아니라 `직무별 렌즈를 바꿔 재배치`하면 된다.

### 2-2. 백엔드 메인 프로젝트로 `BIZ-FUND-AI`가 적합한 이유

`BIZ-FUND-AI`는 README와 docs, 실제 코드 구조를 보면 단순 AI 챗봇이 아니다.

확인된 근거:

- `docs/01_concept_and_requirements.md`
  - 사용자 여정이 5단계로 정리되어 있음
  - 온보딩, 진단, 시뮬레이션, 비교, 사후관리까지 서비스 흐름이 설계되어 있음
- `docs/02_system_architecture.md`
  - FastAPI API 계층
  - PostgreSQL + pgvector
  - BizMong LangGraph 구조
  - 별도 진단 서비스와 시뮬레이션 서비스 분리
  - PolicySyncAgent 파이프라인
- `docs/03_data_design_spec.md`
  - users, businesses, policies, policy_chunks, chat_logs, batch_logs, admin_audit_logs 등 실제 백엔드 설계 근거가 풍부함
- `docs/api_spec/*.md`
  - 인증, 사업장, 정책, 채팅, 진단, 알림, 관리자 API까지 문서화
- 실제 코드 구조
  - `backend/src/app/api/v1/*_router.py`
  - `backend/src/app/domains/*/{service,repository,model,schema}.py`
  - `backend/src/app/agents/biz_mong/*`
  - `backend/src/app/agents/policy_sync_agent.py`
  - `backend/src/app/scripts/evaluate_bizmong_quality.py`
  - `backend/tests/*`

즉 이 프로젝트는 백엔드 포트폴리오에서 말할 수 있는 항목이 많다.

- API 설계
- 도메인 분리
- DB 스키마 설계
- 벡터 검색과 일반 검색 혼합
- 배치/동기화 파이프라인
- 장애 격리
- 평가 자동화
- 운영 관측성

이건 일반 백엔드 포지션에도 충분히 먹힌다.

### 2-3. 솔루션엔지니어 트랙을 따로 만들 수 있는지

처음엔 약할 수 있다고 봤지만, 실제 자료를 보면 완전히 빈약하진 않다.

특히 `BIZ-FUND-AI`에서 솔루션엔지니어 관점으로 쓸 수 있는 재료:

- 문제 정의가 사용자 언어로 잘 잡혀 있음
- 소상공인 사용자 여정이 단계별로 설계되어 있음
- 정책 추천만이 아니라 진단, 시뮬레이션, 비교, 사후관리까지 이어짐
- 관리자 API와 운영 대시보드, 배치 모니터링 구조가 있음
- 비용 절감, 추천 정확도, 사용자 입력 최소화 같은 사업 관점 메시지가 있음
- B2C/B2B 확장 방향과 이해관계자 가치가 문서화되어 있음

다만 한계도 있다.

- `고객 커뮤니케이션`, `도입 제안`, `영업 협업`, `PoC 운영 문서` 같은 전형적 솔루션엔지니어 산출물은 상대적으로 약하다.

그래서 솔루션엔지니어 파트는 아래처럼 잡는 것이 좋다.

- 완전히 별개 직무 포트폴리오처럼 크게 키우지 않는다.
- `문제정의 → 사용자 시나리오 → 기능 설계 → 운영 설계` 관점의 2페이지 정도로 압축한다.
- `기술을 구현한 사람`이 아니라 `문제를 서비스 구조로 번역한 사람`처럼 보이게 만든다.

- 현재 11페이지 구조를 크게 해치지 않으면서 트랙 하나를 더 넣을 수 있다.
- 인쇄/PDF 기준으로도 부담이 과하지 않다.
- `HR Copilot` 4페이지를 3페이지로 압축하고, `BIZ-FUND-AI`는 오히려 5페이지로 강화하는 편이 목적에 맞다.

## 4. 페이지별 상세 계획

### 페이지 1. 표지

목표:

- 기존 `AI 서비스 개발자` 메시지를 `3트랙 통합형`으로 바꾸기

핵심 문구 방향:

- `AI Agent Developer / Backend Developer / Solution Engineer`
- 또는
- `설계하고, 구현하고, 운영 관점까지 연결하는 개발자`

유지할 요소:

- 현재 표지의 KPI 카드 형식
- `HR Copilot 91.8점`
- `BIZ-FUND-AI 15배 절감`
- `평가 하네스 83.3%`

바꿀 요소:

- 메인 타이틀이 너무 에이전트에만 치우쳐 있음
- `LangGraph · LangSmith · RAG` 중심 문구를 유지하되 `FastAPI · PostgreSQL · 운영 설계`가 같이 보이게 조정

### 페이지 2. 포트폴리오 지도

목표:

- 이 문서가 왜 하나의 HTML 안에 3개 직무를 담는지 먼저 설득

구성:

- 왼쪽: 3개 직무 트랙 개요
- 오른쪽: 각 트랙에 어떤 프로젝트가 연결되는지

추천 매핑:

- 에이전트 개발자: `HR Copilot`, `BIZ-FUND-AI`의 BizMong
- 백엔드 개발자: `BIZ-FUND-AI` 전반
- 솔루션엔지니어: `BIZ-FUND-AI`의 문제정의, 사용자흐름, 관리자운영

여기서 중요한 점:

- `HR Copilot = 팀 프로젝트`
- `BIZ-FUND-AI = 1인 기획/설계/구현`

이 대비를 아주 명확하게 보여줘야 한다.

### 페이지 3. 공통 소개

목표:

- 세 트랙이 따로 놀지 않게 연결하는 페이지

핵심 메시지:

- 나는 에이전트만 만드는 사람이 아니라
- `문제를 정의하고`
- `서비스 구조를 설계하고`
- `백엔드와 데이터 흐름을 구현하고`
- `LLM 결과를 검증하는 사람`이라는 흐름

추천 블록:

- 문제정의
- 시스템 설계
- 구현
- 검증/운영

이 페이지는 사실상 이후 9페이지를 읽는 기준점이 된다.

### 페이지 4. 에이전트 개발자 1 - HR Copilot 품질 개선 스토리

목표:

- 에이전트 개발자 강점은 첫 진입부에서 바로 보여주기

### 페이지 6. 에이전트 개발자 3 - 평가와 추적

목표:

### 페이지 7. 백엔드 개발자 1 - BIZ-FUND-AI 서비스 개요

목표:

- 백엔드 파트의 시작은 기술이 아니라 `서비스 범위`부터 보여주기

이 페이지에 들어갈 내용:

- 누구를 위한 서비스인지
- 어떤 문제를 푸는지
- 어떤 기능이 하나의 서비스로 묶였는지

추천 기능 묶음:

- 사업장 온보딩
- 정책 탐색
- AI 상담
- 정밀 진단
- 시뮬레이션
- 알림
- 관리자 운영

핵심 메시지:

- `단일 AI 기능`이 아니라 `서비스 전체를 1인이 설계/구현`했다는 점

### 페이지 8. 백엔드 개발자 2 - 아키텍처 / 도메인 / API

목표:

- 일반 백엔드 면접관이 가장 궁금해할 부분을 정면으로 보여주기

이 페이지 핵심:

- FastAPI API 계층
- 도메인 분리
- router / service / repository / model / schema 구조
- 인증, 사업장, 정책, 채팅, 진단, 알림, 관리자 API

왜 중요한가:

- 이 페이지에서 `AI 데모 개발자`가 아니라 `백엔드 구조를 잡을 줄 아는 개발자`로 보이게 된다.

추천 시각화:

- `Client -> API -> Domain Service -> Repository -> PostgreSQL/pgvector/Elasticsearch`

### 페이지 9. 백엔드 개발자 3 - 데이터 설계와 핵심 로직

목표:

- 단순 CRUD가 아니라 서비스 로직을 담은 데이터 설계라는 점을 보여주기

추천 내용:

- `users`, `businesses`, `business_financial_snapshots`
- `policies`, `policy_chunks`
- `chat_rooms`, `chat_logs`
- `match_logs`, `simulation_logs`
- `batch_logs`, `admin_audit_logs`

함께 보여줄 로직:

- 진단 서비스 분리
- 시뮬레이션 API 분리
- Hybrid RAG
- `thread_id = room_id` 기반 상태 연결

이 페이지의 핵심 문장:

- `AI 기능을 넣기 위해 DB를 붙인 것`이 아니라
- `서비스 운영과 기록 추적까지 고려해 데이터 구조를 설계했다`

### 페이지 10. 백엔드 개발자 4 - 파이프라인 / 안정성 / 운영성

목표:

- 여기서 백엔드 강점을 확실히 끝내기

반드시 넣을 내용:

- PolicySyncAgent
- savepoint 기반 개별 공고 실패 격리
- upsert 기반 멱등성
- 파일 헤더 기반 실제 형식 판별
- pgvector + BM25 + RRF
- 평가 스크립트와 테스트

강조 메시지:

- 정확도뿐 아니라 `중단 없이 오래 도는 시스템`을 고민했다
- `관측 가능성`, `재실행 가능성`, `장애 범위 축소`를 설계했다

이 페이지는 현재 포트폴리오의 페이지 8~10 내용을 백엔드 관점으로 다시 엮으면 된다.

- 사용자 문제 3개
- 왜 기존 방식이 불편한지
- 그 문제를 어떤 기능 묶음으로 풀었는지

이 페이지에서 보여줘야 할 역량:

- 요구사항 해석
- 사용자 언어를 기능 설계로 번역하는 능력
- 우선순위 설정

### 페이지 12. 솔루션엔지니어 2 - 운영 / 관리자 / 확장 전략 / 클로징

목표:

- 솔루션을 만들고 끝이 아니라 운영 구조까지 생각했다는 점 강조

추천 요소:

- 관리자 대시보드
- 배치 로그
- 감사 로그
- 운영 모니터링
- B2C/B2B 확장 가능성
- 어떤 이해관계자에게 어떤 가치가 있는지

이 페이지는 완전한 영업형 솔루션엔지니어 페이지보다는,
`비즈니스 문제를 서비스 운영 구조까지 연결해서 설계한 사람`이라는 인상을 주는 데 집중하는 것이 좋다.

마무리 문구 방향:

- `에이전트만 다룬 것이 아니라, 서비스로 작동하게 만드는 구조를 설계했다`
- `기획, 백엔드, AI 검증을 하나의 흐름으로 연결할 수 있다`

## 5. 기존 페이지에서 무엇을 줄이고 무엇을 늘릴지

### 줄일 부분

- 현재 `HR Copilot` 4페이지는 3페이지로 압축 가능
- `프로젝트 소개성 반복 문구`는 줄이고 구조/개선/평가 중심으로 재편
- `클로징 페이지`의 감성 문구 비중은 약간 축소

### 늘릴 부분

- `BIZ-FUND-AI`는 백엔드 파트로 더 전면에 배치
- 기존엔 AI/RAG 중심이 강했는데, 이제는
  - API
  - 도메인 구조
  - DB
  - 운영 설계
  - 관리자 기능
  를 더 명확히 드러내야 함

### 새로 생길 부분

- 공통 소개 페이지 1장
- 솔루션엔지니어 관점 페이지 2장

## 6. 디자인/서술 원칙

이번 개편에서는 페이지를 직무별로 나누더라도 디자인은 `한 문서 안의 일관성`을 유지해야 한다.

추천 방식:

- 공통 베이스 디자인 유지
- 직무별 색만 다르게
  - 에이전트 개발자: 기존 블루/라임 계열 유지 가능
  - 백엔드 개발자: 차분한 네이비/그레이 기반
  - 솔루션엔지니어: 베이지/그린 또는 운영 대시보드 느낌의 중성톤

서술 원칙:

- 에이전트 파트: `실험`, `검증`, `품질 개선`
- 백엔드 파트: `구조`, `데이터`, `안정성`, `운영`
- 솔루션 파트: `문제정의`, `사용자 흐름`, `기능 전략`, `운영 가치`

## 7. 최종 추천 결론

최종 방향은 아래가 가장 적절하다.

- 3개 직무 트랙을 모두 살린다.
- 다만 비중은 `백엔드 > 에이전트 > 솔루션엔지니어`로 둔다.
- `BIZ-FUND-AI`를 가장 중요한 메인 프로젝트로 올린다.
- `HR Copilot`은 에이전트 역량을 증명하는 대표 사례로 압축한다.
- 솔루션엔지니어 파트는 새 프로젝트를 억지로 만들지 말고 `BIZ-FUND-AI` 안의 기획/운영 관점을 추출해 2페이지로 구성한다.

가장 추천하는 비중은 다음과 같다.

- 에이전트 개발자: 3페이지
- 백엔드 개발자: 4페이지
- 솔루션엔지니어: 2페이지
- 공통 페이지: 3페이지

총 12페이지.

이 구성이 좋은 이유는,
사용자가 어느 포지션으로 보더라도 `한 사람의 강점이 분산되지 않고 오히려 정리되어 보이기 때문`이다.

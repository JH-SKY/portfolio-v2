# 한 HTML 안에 3개의 독립 포트폴리오를 넣는 개편 계획

작성일: 2026-07-02

## 1. 이번 계획의 가장 중요한 기준

이번 작업의 목적은 `요약본 하나`를 만드는 것이 아니다.

정확한 목적은 아래와 같다.

- `에이전트 개발자 지원용 포트폴리오` 1개
- `백엔드 개발자 지원용 포트폴리오` 1개
- `솔루션엔지니어 지원용 포트폴리오` 1개

이 3개를 각각 `독립적으로 제출 가능한 수준`으로 만들고,
파일은 `하나의 HTML` 안에 연속으로 넣는다.

즉 핵심은 이것이다.

- 파일은 하나
- 포트폴리오는 셋
- 페이지는 직무별로 충분히 사용
- 압축하지 않음
- 제출할 때는 필요한 페이지 구간만 PDF로 뽑음

이번 작업에서 가장 하면 안 되는 실수는 `어차피 한 HTML이니까 페이지를 줄이자`는 접근이다.

그건 이번 목적과 완전히 반대다.

사용자는 백엔드 지원할 때 백엔드 구간만 제출한다.
에이전트 지원할 때는 에이전트 구간만 제출한다.
그러므로 각 구간은 `그 직무에 내는 완성형 포트폴리오`처럼 충분한 분량을 가져야 한다.

## 2. 잘못 잡으면 안 되는 전제

이번 계획에서는 아래 전제를 버려야 한다.

- 한 HTML이므로 전체 페이지 수를 줄여야 한다
- 직무별 포트폴리오를 압축해야 한다
- 세 직무가 한 문서 안에서 균형 있게 짧게 공존해야 한다

이 전제는 틀렸다.

맞는 전제는 아래다.

- 한 HTML은 그냥 `보관/관리 방식`일 뿐이다
- 실제 제출 단위는 `직무별 페이지 구간`이다
- 따라서 각 구간은 `그 자체로 긴 포트폴리오`여도 상관없다
- 오히려 압축하면 지원 직무에서 설득력이 떨어질 수 있다

## 3. 최종 구조 원칙

구조는 이렇게 간다.

- HTML 파일 1개
- 그 안에 `포트폴리오 A`, `포트폴리오 B`, `포트폴리오 C`가 순서대로 들어감
- 각 포트폴리오는 표지부터 클로징까지 `자기 완결형`
- 중간에 다른 직무 내용이 끼어들지 않음

즉 실제 읽는 사람 입장에서는:

- 에이전트용 구간만 보면 하나의 에이전트 포트폴리오
- 백엔드용 구간만 보면 하나의 백엔드 포트폴리오
- 솔루션용 구간만 보면 하나의 솔루션 포트폴리오

처럼 느껴져야 한다.

## 4. 페이지 수에 대한 기준

이번에는 `직무별 충분한 분량`을 우선한다.

### 에이전트 개발자용

현재도 이미 10장 안팎의 자산이 있다.
따라서 `10장 이상` 유지해도 전혀 문제 없다.

오히려 이 포트폴리오는 지금까지 가장 공들여 만든 축이므로,
굳이 줄일 이유가 없다.

### 백엔드 개발자용

`BIZ-FUND-AI`는 기획, 설계, API, DB, 배치, 운영까지 설명 가능한 프로젝트이기 때문에
이것도 `10장 이상`으로 충분히 길게 가져가는 것이 맞다.

4장, 5장으로 줄이면 오히려 강점이 사라질 가능성이 크다.

### 솔루션엔지니어용

이쪽은 상대적으로 재료가 적을 수는 있다.
그래도 `짧게 요약`부터 생각할 게 아니라,
먼저 쓸 수 있는 내용을 최대한 다 펼쳐보고
그 다음 분량을 판단해야 한다.

즉 기준은:

- 에이전트: 줄이지 않음
- 백엔드: 줄이지 않음
- 솔루션: 부족하면 그때 조정, 처음부터 압축하지 않음

## 6. 권장 전체 페이지 구조

여기서는 `줄이는 안`이 아니라 `충분히 쓰는 안`으로 잡는다.

### 권장안

- 1~11페이지: 에이전트 개발자용 포트폴리오
- 12~22페이지: 백엔드 개발자용 포트폴리오
- 23~30페이지: 솔루션엔지니어용 포트폴리오

총 30페이지

이 안이 지금 기준에서 가장 자연스럽다.

이유:

- 현재 에이전트용 포트폴리오 자산이 이미 많다
- 백엔드는 `BIZ-FUND-AI` 하나만으로도 10장 이상 설명 가능하다
- 솔루션은 상대적으로 짧을 수 있지만 6~8페이지 정도는 충분히 만들 수 있다

즉 `30페이지`가 길어 보여도 문제 없다.
어차피 제출은 구간별로 하기 때문이다.

## 10. 디자인 원칙

파일은 하나지만 구간은 세 개이므로,
디자인도 `완전히 같은 문서의 반복`처럼 보이면 안 된다.

### 에이전트 구간

- 현재 포폴 스타일을 많이 재사용
- 실험 / 추적 / 개선 느낌 강화

### 백엔드 구간

- 더 구조적이고 단정한 느낌
- 아키텍처 / 표 / 데이터 흐름 중심

### 솔루션 구간

- 사용자 여정 / 가치 제안 / 운영 흐름 중심
- 제안서 느낌이 나도록 구성

단, 공통 CSS 자산은 재사용 가능하게 설계한다.

## 11. 실제 구현할 때의 편집 원칙

### 하지 말아야 할 것

- 백엔드 구간을 4장으로 줄이기
- 에이전트 구간을 억지로 요약하기
- 솔루션 구간을 시작도 하기 전에 축약 가정하기
- 세 직무 내용을 한 중간 페이지에 섞기

### 해야 할 것

- 각 구간을 `독립 제출본`처럼 설계하기
- 구간마다 별도 표지 두기
- 구간마다 별도 클로징 두기
- 페이지 번호는 전체 기준으로 가되, 구간 식별 라벨을 따로 넣기

예시:

- `Agent Portfolio / p.01`
- `Backend Portfolio / p.12`
- `Solution Portfolio / p.23`

## 12. 최종 결론

이번 작업은 요약본 제작이 아니다.

정확히는:

- `긴 에이전트 포트폴리오`
- `긴 백엔드 포트폴리오`
- `상대적으로 짧을 수 있지만 독립 제출 가능한 솔루션 포트폴리오`

이 3개를 하나의 HTML에 넣는 작업이다.

따라서 가장 중요한 결론은 아래다.

- 압축하지 않는다
- 각 직무용 포트폴리오를 따로 제출하는 전제로 만든다
- 한 HTML은 단지 관리 방식일 뿐이다
- 분량보다 중요한 것은 `직무별 완결성`이다

에이전트 설계 흐름
왜 지원자 리스크를 먼저 뽑았는가 공백기, 직무 전환, 성과 부풀리기처럼 면접에서 반드시 확인해야 할 리스크가 질문에서 빠지지 않도록 하기 위해서입니다.
왜 역할을 나눴는가 각 단계의 책임을 명확히 하고 품질 저하를 줄이기 위해서입니다.
어떻게 운영 관점까지 봤는가 재생성 루프와 실행 추적 로그를 보며 품질 저하 원인을 추적했습니다.
LLMOPS OBSERVABILITY

LangSmith Trace
품질 저하 원인 추적

TEAM PROJECTHR Copilot

LangSmith Trace로 노드별 중간 상태를 추적해 품질 저하 원인을 찾고, 구조 개선으로 이어갔습니다.

결과 로그만으로는 품질 개선이 어렵습니다 질문이 생성됐다는 사실만으로는 왜 품질이 흔들렸는지 설명할 수 없었습니다.
중간 상태 추적 실행 기록 기준으로 노드별 상태 변화를 직접 추적했습니다.
원인을 특정하고 구조를 개선 Trace에서 확인한 품질 저하 지점을 verification_point_extractor와 최종 선별 로직 개선으로 연결했습니다.
추적 기반 원인 분석
LLMOps 개선 조치 verification_point_extractor 추가, 필수 질문 보호 단위 세분화, 최종 선별 기준 강화를 통해 관측 결과를 운영 개선으로 연결했습니다.
추적 결과 1. 재생성 여부보다 피드백 전달 구조가 핵심이었습니다 reviewer는 계속 수정 필요를 반환했지만, 질문 생성 이후 feedback state가 초기화되어 driller 단계가 같은 패턴으로 되돌아가는 것을 확인했습니다.
추적 결과 2. 생성 성공과 최종 반영은 다른 문제였습니다 verification_point_extractor가 추출한 필수 질문이 questioner 단계에서는 살아 있었지만, 최종 선별 단계에서 빠지는 문제를 확인했습니다.
Trace를 품질 저하 원인을 추적하는 운영 도구로 사용했습니다. 결과 로그만 보면 질문이 만들어졌다만 보입니다. LangSmith Trace를 추적하여 각 노드의 STATE 값과 전달 여부를 직접 확인할 수 있었고, 이를 기준으로 원인 지점을 좁힐 수 있었습니다.

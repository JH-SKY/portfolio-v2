# 포트폴리오 계획서 근거 기반 점검

작성일: 2026-07-02

대상 문서: `백엔드_비즈니스_포트폴리오_구성계획_2026-07-02.md`

점검 원칙:

- 실제 코드, README, docs, 테스트 파일에서 확인되는 내용만 포트폴리오에 넣는다.
- 코드 근거가 약한 내용은 "구현했다"가 아니라 "설계/제안/향후 PoC 운영 방식"으로 분리한다.
- 채용 담당자나 면접관이 바로 물을 질문, 즉 "근거가 뭐냐", "네 역할이 뭐냐", "장애나 품질은 어떻게 다뤘냐", "테스트는 있냐"에 답할 수 있는 항목을 우선한다.
- 기존 `index.html` 에이전트 개발자 포트폴리오는 건드리지 않는다.

---

## 0. 결론

현재 계획서는 방향은 좋지만, 실제 코드에 존재하는 강한 근거 몇 개가 덜 강조되어 있다.

특히 백엔드 포트폴리오에는 다음을 더 세게 넣어야 한다.

- `biz-fund-ai`의 PostgreSQL Advisory Lock 기반 스케줄러 단일 실행 보장
- `biz-fund-ai`의 AgentRunLog/AgentNodeLog 기반 관리자 관측성
- `biz-fund-ai`의 섹션 기반 chunking, contextual embedding, SHA-256 content hash 기반 임베딩 스킵
- `biz-fund-ai`의 국세청 사업자 검증, 정책 추천, 검색 품질 테스트
- `hr-copilot`의 채용공고 컴플라이언스 분석 파이프라인과 실험 실행 기능
- `hr-copilot`의 LLM 노드별 trace, 비용 집계, 관리자 대시보드
- `hr-copilot`의 ZIP 안전 처리와 일괄 문서 등록 사전 선별 구현

반대로 FDE/비즈니스 엔지니어 포트폴리오에서 "고객 미팅", "프로젝트 리딩", "제휴 모델 검증"처럼 실제 실행 증거가 없는 표현은 조심해야 한다. 이런 내용은 과거 성과가 아니라 "고객 PoC를 이렇게 운영하겠다"는 직무 적합성 페이지로 분리해야 한다.

## 1. 백엔드 포트폴리오에서 빠진 실제 강점

### 1.1 PostgreSQL Advisory Lock 기반 스케줄러 단일 실행 보장

현재 계획서에는 배치/트랜잭션 안정성이 들어가 있지만, 실제 코드에 있는 가장 강한 운영 안정성 근거가 빠져 있다.

근거 파일:

- `biz-fund-ai/backend/src/app/worker/scheduler.py`

확인된 구현:

- APScheduler 기반 일일 정책 동기화 작업
- `pg_try_advisory_lock(:lock_id)`로 스케줄러 단일 실행 보장
- 여러 서버 인스턴스가 떠도 락을 얻은 인스턴스만 scheduler를 start
- `RUN_SCHEDULER=false`로 로컬/특정 환경에서 scheduler 비활성화 가능
- 일일 작업에서 BizInfo 정책 동기화, AI 구조화, 벡터 임베딩까지 연결

포트폴리오 반영 방식:

- 백엔드 포트폴리오에 "Batch Reliability / Multi-instance Scheduler" 페이지를 추가한다.
- 단순히 "스케줄러 사용"이 아니라 "수평 확장 환경에서 중복 배치 실행을 막기 위해 DB advisory lock을 사용했다"로 말해야 한다.
- 면접에서 어필할 문장:
  - "APScheduler만 쓰면 멀티 인스턴스 환경에서 같은 배치가 중복 실행될 수 있어 PostgreSQL Advisory Lock으로 leader instance만 작업을 실행하게 설계했습니다."

중요도: 매우 높음

### 1.2 AgentRunLog/AgentNodeLog 기반 AI 실행 관측성

현재 계획서에는 observability가 들어가 있지만, 실제 구현 수준에 비해 약하게 적혀 있다. 이건 백엔드 면접관에게 꽤 강한 포인트다.

근거 파일:

- `biz-fund-ai/backend/src/app/domains/admin/service.py`
- `biz-fund-ai/backend/src/app/agents/biz_mong/telemetry.py`
- `biz-fund-ai/backend/src/app/api/v1/admin_router.py`
- `biz-fund-ai/docs/03_data_design_spec.md`

확인된 구현:

- `monitoring_agent_overview`
  - 전체 실행 수
  - 성공률
  - 평균/p50/p95 latency
  - input/output token
  - total cost USD/KRW
  - fallback run 수
  - CTA click
  - dislike feedback count
  - fallback rate, dislike feedback rate
  - intent/model/prompt version/graph version/RAG strategy version별 breakdown
- `monitoring_agent_nodes`
  - node별 실행 수
  - avg/p95 latency
  - token/cost
  - error count/error rate
- `monitoring_agent_runs`
  - run 목록
  - session id, route intent, final agent, fallback mode, RAG hit count
  - prompt/model/graph/rag strategy version
- `monitoring_agent_run_detail`
  - run 상세
  - node log
  - CTA event
  - error code/message
- `telemetry.py`
  - model별 가격 테이블
  - cost estimation
  - node log builder

포트폴리오 반영 방식:

- 백엔드 포트폴리오에는 "AI Observability & Admin Monitoring" 페이지를 별도로 둔다.
- FDE 포트폴리오에는 "고객 PoC 운영 시 어떤 지표로 품질/비용/전환을 설명할 수 있는가"로 재활용한다.
- 단순 로그가 아니라 "운영자가 의사결정할 수 있는 지표"로 표현한다.

중요도: 매우 높음

### 1.3 정책 RAG 파이프라인의 chunking, contextual embedding, hash skip

현재 계획서에 Hybrid RAG가 있지만, 실제 `PolicyEmbeddingService`의 설계 판단이 더 구체적이다.

근거 파일:

- `biz-fund-ai/backend/src/app/domains/policy/embedding_service.py`

확인된 구현:

- 섹션 헤더 기반 chunking
  - 지원대상
  - 지원내용
  - 신청방법
  - 제출서류
  - 문의처 등 정책 문서 구조를 기준으로 분리
- 섹션이 없을 때 sliding window fallback
  - 800자 window
  - 100자 overlap
- contextual embedding
  - embedding 입력에 `[정책명]`, `[기관]`, `[지원유형]` prefix를 붙임
  - DB에는 원문 chunk를 저장
- SHA-256 content hash
  - 정책 내용이 변하지 않으면 재임베딩을 skip
  - OpenAI API 비용과 DB write를 줄임
- OpenAI embedding batch 처리
- 개별 정책 처리 실패 시 savepoint로 격리

포트폴리오 반영 방식:

- "Hybrid RAG" 페이지 안에 반드시 실제 설계 판단으로 넣는다.
- 면접에서 어필할 문장:
  - "정책 문서는 문단보다 섹션 의미가 중요해서 단순 고정 길이 chunking이 아니라 지원대상/지원내용/신청방법 같은 도메인 헤더 기반 chunking을 먼저 적용했습니다."

중요도: 매우 높음

### 1.5 실제 테스트로 증명 가능한 검색/추천/검증 품질

현재 계획서의 테스트 파트가 조금 일반적이다. 실제 테스트명과 검증 범위를 보여주면 훨씬 설득력이 있다.

근거 파일:

- `biz-fund-ai/backend/tests/domains/policy/test_policy_service.py`
- `biz-fund-ai/backend/tests/domains/policy/test_rule_based_match_engine.py`
- `biz-fund-ai/backend/tests/domains/business/test_biz_verification.py`

확인된 테스트 범위:

- Elasticsearch chunk index에서 nori analyzer 사용 검증
- analyzer token 추출 로직 검증
- chunk id를 검색 문서 primary key처럼 사용하는 직렬화 검증
- sparse search ranking 검증
- dense+sparse, sparse_only 등 retrieval backend label 검증
- 정책 추천에서 사업자 ID 권한/컨텍스트 검증
- bookmark 상태 포함/토글 검증

  - 휴업자/폐업자 invalid
  - 미등록
  - empty data
  - timeout
  - HTTP error
  - API key 없음
  - 수동 skip
  - 이미 검증된 사업자 skip

포트폴리오 반영 방식:

- "Testing & Quality Gate" 페이지에서 테스트 개수보다 테스트가 막는 실제 리스크를 보여준다.
- 예:
  - "외부 API 장애/타임아웃/키 누락이 회원가입과 사업자 검증 흐름을 깨지 않도록 테스트"
  - "검색 품질은 nori analyzer, sparse ranking, retrieval backend label까지 테스트"

중요도: 매우 높음

## 2. FDE/비즈니스 엔지니어 포트폴리오에서 빠진 실제 강점

### 2.1 채용공고 컴플라이언스 분석은 FDE 포트폴리오의 핵심 사례가 되어야 한다

현재 계획서에서는 HR Copilot의 질문 생성 품질 개선이 중심이다. 하지만 FDE 관점에서는 채용공고 컴플라이언스 분석이 더 직무 적합도가 높다.

근거 파일:

- `hr-copilot/backend/services/job_posting_service.py`
- `hr-copilot/backend/services/job_posting_trace_service.py`
- `hr-copilot/docs/기획/채용공고분석데이터흐름.md`
- `hr-copilot/docs/기획/채용공고_RAG_고도화가이드.md`

왜 강한가:

- 고객사의 실제 리스크를 기능 요구사항으로 변환한 사례다.
- 법/규정/운영 리스크를 rule, evidence, trace, experiment로 제품화했다.
- 단순 개발이 아니라 "현업 담당자가 왜 이 공고를 고쳐야 하는지 설명 가능한 분석"에 가깝다.

포트폴리오 반영 방식:

- FDE 포트폴리오 첫 번째 핵심 프로젝트를 `BizMong` 추천 PoC로만 두지 말고, `HR Copilot 채용공고 컴플라이언스`를 동급 또는 더 강한 사례로 둔다.
- 페이지 제목 예:
  - "고객 리스크를 제품 요구사항으로 번역한 채용공고 컴플라이언스 PoC"

중요도: 매우 높음

### 2.2 Experiment run 기능은 PoC 검증 역량의 직접 증거다

FDE/세일즈 엔지니어 면접관은 "PoC 성공 기준을 어떻게 잡고 검증했는가"를 묻는다. `hr-copilot`에는 실험 실행 구조가 실제 코드에 있다.

근거 파일:

- `hr-copilot/backend/services/job_posting_service.py`
- `hr-copilot/backend/api/v1/routers/job_posting_router.py`

확인된 구현:

- experiment run 생성
- experiment run 목록/상세 조회
- experiment job 제출
- active experiment job 조회
- experiment case result 저장
- pipeline version과 analysis version 관리

포트폴리오 반영 방식:

- "PoC 검증" 페이지에 `biz-fund-ai`의 평가셋뿐 아니라 `hr-copilot`의 experiment run도 넣는다.
- 좋은 문장:
  - "기능 시연에서 끝나지 않고, 분석 pipeline version과 case result를 남겨 반복 실험이 가능하도록 설계했습니다."

중요도: 매우 높음

### 2.3 일괄 문서 등록과 사전 선별은 구현된 고객 업무 자동화다

현재 계획서에 "일괄등록 사전 선별 설계"가 들어가 있지만, 실제 구현이 있다는 점을 더 강하게 말해야 한다.

근거 파일:

- `hr-copilot/backend/services/document_bulk_import_service.py`

확인된 구현:

- ZIP 또는 다중 파일 업로드 처리
- ZIP member path 안전 검사
  - absolute path 차단
  - `..` path 차단
- 파일명/폴더 기반 문서 grouping
- 파일명 신뢰도가 낮을 때 문서 내용의 email/phone 기반 grouping 보정
- 허용 확장자 검증
- OpenAI key가 있으면 LLM profile extraction
- 없으면 heuristic fallback
- screening preview 생성
  - document readiness
  - profile completeness
  - job fit
  - evidence quality
  - risk adjustment
  - recommendation
  - suggested action
  - interview focus
- 민감 정보가 screening basis에 들어가지 않도록 warning 처리

포트폴리오 반영 방식:

- FDE 포트폴리오에서 "현업 채용 담당자의 반복 업무를 줄이는 업무 자동화" 사례로 넣는다.
- 백엔드 포트폴리오에서는 파일 처리 보안과 데이터 grouping 전략으로 넣는다.

중요도: 매우 높음

### 4.2 "프로젝트 리딩" 표현

판정: 일부 조심

이유:

- 문서와 코드로 설계/구현/검증 역량은 강하게 보인다.
- 하지만 팀 리딩, 일정관리, 이해관계자 관리에 대한 직접 증거는 제한적이다.

수정:

- "프로젝트를 리딩했다" 대신:
  - "요구사항을 기능/데이터/운영 지표로 구체화했다"
  - "PoC 검증 구조를 설계했다"
  - "운영자가 확인할 수 있는 관리자 지표를 구현했다"

---

### 4.3 "LangSmith API 연동"처럼 보이는 표현

판정: 표현 조정 필요

이유:

- 문서에는 LangSmith/trace 관련 내용이 있지만, 현재 확인한 실제 코드에서는 자체 `LlmCallLog`, `JobPostingTraceRecorder`, trace id 기반 기록이 더 확실하다.

수정:

- "LangSmith로 해결했다"라고 단정하지 않는다.
- 실제 코드 기준으로는 "trace id와 LLM call log로 노드별 실행을 기록했다"가 안전하다.
- 문서 기반으로 언급할 경우 "문서화된 LangSmith trace 분석"처럼 증거 수준을 낮춘다.

## 7. 포트폴리오에 넣을 증거 표기 방식

각 페이지 하단에 작은 Evidence 블록을 두는 것을 추천한다.

예:

```text
Evidence
- biz-fund-ai/backend/src/app/worker/scheduler.py
- biz-fund-ai/backend/src/app/domains/policy/embedding_service.py
- biz-fund-ai/backend/tests/domains/policy/test_elasticsearch_helpers.py
```

이렇게 하면 지어낸 느낌이 줄고, 면접관이 "이거 실제로 했냐"라고 물어도 바로 답할 수 있다.

단, 포트폴리오 화면 자체가 너무 개발 문서처럼 보이지 않게 해야 한다. Evidence는 작은 caption 또는 expandable 영역으로 처리하고, 본문은 문제-판단-구현-효과 순서로 보여주는 것이 좋다.

## 9. 다음 포트폴리오 제작 시 적용할 규칙

1. 모든 강점 문장에는 근거 파일을 붙인다.
2. 구현 근거가 없는 문장은 과거 성과로 쓰지 않는다.
3. 백엔드 포트폴리오는 기술명보다 운영 리스크 해결을 먼저 보여준다.
4. FDE 포트폴리오는 기능보다 고객 질문에 대한 답을 먼저 보여준다.
5. 같은 프로젝트라도 백엔드와 FDE에서 어필 포인트를 다르게 쓴다.
6. 각 포트폴리오에는 "Evidence Index" 페이지를 넣어 신뢰도를 높인다.
7. "내가 무엇을 만들었는지"보다 "왜 그렇게 설계했는지"를 더 크게 보여준다.

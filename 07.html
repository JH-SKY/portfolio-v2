PROJECT 1 · TROUBLE POINT

문제는 Prompt가 아니라
State 유실이었습니다.

LLM이 이상하다고 결론내기 전에, 어디서 정보가 끊겼는지 먼저 확인했습니다.

Detected Symptom

품질 점수가 낮게 유지

Retry는 실행되지만 개선 방향이 다음 노드에 제대로 반영되지 않았습니다.

Root Cause

feedback state 초기화

Reviewer의 피드백이 후속 Questioner까지 전달되지 않아 같은 실수가 반복됐습니다.

1
Retry 로직 확인

분기 실행은 정상. 재시도 자체가 문제는 아니었습니다.

2
Reviewer 기준 확인

프롬프트 기준을 조정했지만 주요 원인은 아니었습니다.

3
State 흐름 전수 분석

LangSmith trace를 따라가며 feedback state 유실 지점을 발견했습니다.

Reviewer feedback 생성
→ Questioner feedback 반영 시도
→ feedback State 초기화
→ 후속 노드 개선 방향 손실

해결: State 초기화 시점 조정 + feedback 전달 구조 재설계
“LLM이 이상하다”가 아니라 “어느 state에서 정보가 끊겼는지”를 보기 시작했습니다.

11
DEEP DIVE · BATCH PIPELINE

1,255건을 저장했는데
실제 DB에는 일부만 남았습니다.

로그의 success_cnt와 실제 DB 상태가 다르다는 것을 확인하며 원인을 좁혔습니다.

Before

try-except + rollback

중복 키가 발생할 때 rollback을 호출했고, 마지막 commit 이후 성공 데이터까지 모두 되돌아갔습니다.

After

UPSERT

중복을 예외로 보지 않고 정상 저장 흐름으로 처리했습니다. 동일 작업을 반복해도 결과가 같도록 만들었습니다.

stmt = insert(Policy).values(policy_list)
upsert_stmt = stmt.on_conflict_do_update(
  index_elements=['policy_id'],
  set_={'title': stmt.excluded.title,
       'content': stmt.excluded.content,
       'updated_at': datetime.now()}
)
await session.execute(upsert_stmt)
await session.commit()
Learned

Atomicity

rollback은 한 건만 되돌리는 것이 아니라 마지막 commit 이후 변경을 되돌립니다.

Learned

Idempotency

배치 작업은 여러 번 실행해도 결과가 같아야 안정적입니다.

Learned

Cross Check

로그와 실제 DB 결과를 반드시 교차 검증해야 합니다.

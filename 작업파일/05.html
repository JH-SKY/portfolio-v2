내 역할 AI Agent 개발 · 팀 리딩 · LangGraph 설계 · LangSmith 기반 품질 개선
핵심 질문 좋은 질문을 많이 만드는 게 아니라, 채용 리스크를 검증하는 질문을 만들 수 있는가
“면접 질문은 친절한 문장이 아니라, 리스크를 확인하는 도구여야 한다.” HR Copilot에서 제가 잡은 문제의 기준입니다.

처음부터 목표를 “그럴듯한 질문 생성”으로 두지 않았습니다. 지원자 문서 안의 성과 과장 가능성, 실제 기여 범위, 직무 전환 사유, 협업 태도, 이직 패턴처럼 면접관이 확인해야 할 지점을 질문으로 바꾸는 것이 핵심이었습니다.

why langgraph

한 번에 생성하면, 어디서 망가졌는지 알 수 없었습니다.

그래서 질문 생성을 하나의 프롬프트로 밀어 넣지 않고, 검증 포인트 추출부터 리뷰와 재시도까지 노드로 분리했습니다.

01 prepare context 지원자 문서와 직무 정보 정리
02 verification point 리스크와 검증 포인트 선추출
03 question / answer 질문 후보, 예상 답변, 꼬리 질문 생성
04 review / retry 루브릭 평가와 최대 3회 재생성
05 select top 5 후보 10개 중 최종 질문 선별
experience 01 / state bug

낮은 점수의 원인은 LLM이 아니라, 끊긴 state였습니다.

최종 질문 품질 점수가 계속 낮게 나왔고, 저는 세 가지 가설을 세웠습니다. retry 미작동, reviewer 기준 과도함, 그리고 feedback state 유실. LangSmith trace로 하나씩 제거한 끝에 state 초기화 시점 문제가 원인임을 확인했습니다.

실제 자료가 들어갈 자리: LangSmith trace 캡처, 낮은 점수 예시, 실패 패턴 표 중 1개.
experience 02 / quality design

질문 문장을 고친 게 아니라, 질문이 만들어지는 순서를 바꿨습니다.

BEFORE

문서 전체에서 바로 질문 생성

  • 겉보기에는 자연스러운 질문이 생성됨
  • 눈에 잘 보이는 경험 위주로 질문이 쏠림
  • 성과 과장, 기여 범위, 공백기 같은 리스크가 약하게 반영됨
  • 프롬프트가 커질수록 디버깅 지점이 흐려짐
AFTER

검증 포인트를 먼저 추출한 뒤 질문 생성

  • 지원자별 리스크 신호를 먼저 정리
  • 질문 후보 10개 생성 후 reviewer가 비교
  • 상위 5개만 최종 질문으로 선별
  • 노드별 책임이 분리되어 품질 관리와 추적이 쉬워짐

HR-Copilot

AI 기반 채용 자동화 에이전트 · 4인 팀

GitHub 배포 Notion
LangGraph LangSmith FastAPI PostgreSQL OpenAI API Docker
Problem

문제 인식

HR 담당자는 수십 장의 서류를 일일이 검토해야 하는 반복 작업에 시간을 빼앗깁니다. 면접 질문도 매번 새로 만들어야 합니다.

Approach

선택과 이유

LangGraph를 선택한 것은 노드-엣지 구조로 에이전트 흐름을 명시적으로 설계하고, LangSmith로 각 단계를 추적할 수 있기 때문입니다.

Challenge

핵심 난관

리뷰어가 낮은 점수를 줘도 질문 품질이 개선되지 않는 문제. 프롬프트 수정으로 해결되지 않아 50시간 이상 Trace를 분석했습니다.

Result

결과

질문 품질 점수 78.6 → 91.8점. 마감 10일 전 MVP 완성 후 고도화에 집중할 시간 확보.

🔍 경험 1. LangSmith Trace로 State 유실을 찾아낸 디버깅

품질 개선 핵심

리뷰어가 낮은 점수를 줘도 최종 결과에 문제 있는 질문이 그대로 포함됐습니다. 처음엔 리트라이 로직 버그라고 생각했지만, LangSmith를 켜서 확인하니 3번 리트라이는 정상적으로 동작하고 있었습니다. 그 다음엔 리뷰어 프롬프트가 너무 엄격한 게 아닐까 해서 프롬프트를 여러 번 고쳤지만 결과는 달라지지 않았습니다.

결국 수백 개의 Trace를 하나하나 따라가며 각 노드의 입력과 출력을 비교했습니다. 원인은 State 유실이었습니다. 처음 설계할 때 "질문 생성 노드"만 생각하고 피드백 반영 후 State를 초기화하는 구조로 만들어 뒀기 때문에, 나머지 노드들은 리뷰어 피드백을 전달받지 못한 채 이전 결과를 거의 그대로 재생성하고 있었던 것입니다.

배운 것: LangGraph에서는 노드를 잘 나누는 것만큼이나, State가 어떤 노드에서 생성되고 어디까지 유지되며 어떤 노드가 참조하는지를 처음부터 설계하는 것이 중요합니다. 프롬프트 문제가 아니라 시스템 구조 문제였습니다.

🎯 경험 2. "좋은 면접 질문"을 다시 정의하고 구조에 반영한 경험

도메인 + 기술 결합

State 유실을 해결한 뒤에도 질문을 보면서 "나쁘지는 않은데, 방향이 이상하다"는 느낌이 있었습니다. 직무 경험 확인이나 실무 역량 검증 질문은 잘 나왔지만, 실제 채용 관점에서 더 중요한 것들이 빠져 있었습니다. 잦은 이직, 공백기, 짧은 재직 기간처럼 서류상에서 면접관이 반드시 확인해야 할 리스크 포인트들이었습니다.

LLM에게 "더 좋은 질문을 만들어줘"라고 하는 대신, HR 도메인에서 좋은 면접 질문이 무엇인지를 다시 정의하고, 그 기준을 LangGraph 구조 안에 노드와 평가 흐름으로 반영했습니다. 지원자 서류에서 리스크 포인트를 추출하는 전용 노드를 추가하고, 후보 질문 10개를 먼저 생성한 뒤 서류 기반성·구체성·리스크 검증 가능성·활용도 기준으로 5개를 선택하는 리랭킹 구조를 적용했습니다.

배운 것: 에이전트 품질 개선은 프롬프트 튜닝 전에 "이 시스템이 무엇을 잘해야 하는가"라는 도메인 정의가 먼저입니다. 기준을 구조로 만드는 것이 핵심이었습니다.

📋 경험 3. 문서화와 WBS로 마감 10일 전 MVP를 완성한 팀 리더십

팀 리더 · 프로세스 설계

비전공자 비중이 높은 팀이었기 때문에, 기능을 바로 만들기 시작하면 "이 기능이 맞는지", "API는 어떻게 주고받을지"를 두고 병목이 생길 것이 예상됐습니다. 그래서 초반에 기획안, WBS, API 명세, ERD를 먼저 정리하고, 이 문서를 기준으로 개발 방향과 우선순위를 고정했습니다. 의견 충돌이 생겼을 때도 말로 설득하지 않고, 각자 근거 자료를 가져와 논의하는 원칙을 세웠습니다.

영업 팀장 시절처럼 일 단위로 목표를 세분화하고 하루에도 수시로 진행 상황을 확인했습니다. 팀원들이 스스로 정한 목표치를 초과하기 위해 자발적으로 야근을 선택했고, 그 결과 마감 10일 전에 MVP 1차를 완성했습니다. 확보된 시간을 LangGraph 실험과 고도화에 투자해 품질 점수를 91.8점까지 끌어올렸습니다.

배운 것: 기술만큼이나 팀이 같은 방향을 바라보게 만드는 소통 구조가 중요합니다. "개발 속도"는 코딩 속도가 아니라 의사결정 속도에서 나옵니다.

Problem

문제 인식

소상공인은 자신에게 맞는 정책자금을 찾기 어렵습니다. 공고는 수시로 바뀌고, 조건이 복잡해 직접 검토하기 힘듭니다.

Architecture

구조 설계

Router-Diagnosis-RAG-Stats-Simulation 5개 노드로 역할을 분리해 "하나의 LLM이 모든 걸 처리"하는 블랙박스를 제거했습니다.

Challenge

핵심 난관

Router 노드 설계. 실제 사용자 질문은 복합 의도를 담고 있어, 단순 키워드 기반 라우팅은 복합 질문에서 틀린 노드로 가는 문제가 있었습니다.

Result

결과

추천·진단·RAG·시뮬레이션 4가지 기능이 하나의 상담 흐름 안에서 동작. 정책 공고 자동 수집으로 수작업 등록 제거.

🔀 왜 5개 노드로 분리했나 — 구조 설계의 이유

아키텍처 결정

하나의 LLM이 모든 것을 처리하게 만들 수도 있었습니다. 하지만 그렇게 하면 답변이 왜 그렇게 나왔는지 추적하기 어렵고, 특정 기능만 개선하기도 어렵습니다. Router는 질문 의도를 판단하고, Diagnosis는 사업자 조건을 분석하고, RAG는 정책 문서를 검색하고, Stats는 통계성 정보를 처리하고, Simulation은 예상 결과를 계산합니다.

이렇게 책임을 나누면 문제가 생겼을 때 어느 노드에서 개선해야 하는지 바로 파악할 수 있습니다. 추후 기능을 추가할 때도 전체 구조를 갈아엎지 않고 특정 노드만 확장할 수 있습니다. HR-Copilot에서 State 유실로 겪은 경험이 처음부터 노드 책임 범위를 명확히 설계하도록 만든 직접적인 이유이기도 했습니다.

핵심 판단: 복잡한 것을 단순하게 만드는 가장 좋은 방법은 책임을 잘게 나누는 것입니다. 각 노드가 "자기 일만 잘하면" 전체가 안정적으로 동작합니다.

🔍 Hybrid RAG(RRF)를 선택한 이유와 검증 방법

RAG 고도화

초기 단일 벡터 검색에서는 사용자가 "창업 지원금"이라고 물어볼 때 문서에 "초기창업패키지"나 "사업화 자금"처럼 다른 표현으로 기재된 경우 관련 문서를 찾지 못하는 일이 있었습니다. 벡터 검색은 의미적 유사성에 강하지만 정확한 키워드 매칭에 약하고, 키워드 검색은 그 반대입니다.

두 검색 결과를 RRF(Reciprocal Rank Fusion)로 재정렬하는 Hybrid RAG를 구현했습니다. 검증은 정책/지원금 관련 질문 세트를 사전에 준비하고, 단일 RAG와 하이브리드 RAG에 동일한 질문을 넣은 뒤 관련 문서 상위 포함 여부, 지원 대상·금액·조건 같은 핵심 정보 누락 여부, 출처 기반 답변 가능 여부를 기준으로 비교했습니다.

핵심 개선: 표현이 다른 질문에도 관련 문서를 안정적으로 찾아오는 것이 가장 큰 변화였습니다. RAG의 품질은 검색 단계에서 이미 절반이 결정됩니다.
